## Findings (severity ranked, dedupe된 unified list)

### 🔴 critical (반드시 spec 수정)

**F1. `claim` field origin contradiction** [Opus #1, Codex adv #5, Codex rev §6.2-lossy 관련]
- 위치: §7.1 Step A 매핑 표
- 문제: 5 memory type 중 4개(failure-case / pattern / coding-style / architecture-decision) 에서 Step A가 `claim`을 채우지 않음. §7.2는 "Step A가 채운 필드 LLM이 덮어쓰지 않음" 명시 → fallback path (§7.2 on-failure: candidate) 가 발동하면 `claim`이 빈 상태로 persist → §6.2 `dedupe_key = sha256(type + "|" + normalize(claim))` 가 모든 fallback card 에 대해 동일 키 → 모든 fallback card가 1개로 병합됨.
- 수정: §7.1 매핑 표에 모든 memory type 별 deterministic `claim` source 추가 (예: `failure-case: finding.title || finding.summary`, `pattern: slice.title || slice.id`, `coding-style: drift_finding.title`, `architecture-decision: page.frontmatter.title || page.title`)

**F2. `avoid_when` ↔ `non_applicability` field name mismatch** [Opus #2]
- 위치: §9.1, §9.2 vs §6.1
- 문제: brief 출력은 `avoid_when` key 사용, card schema 는 `non_applicability`. §9 "비어 있으면 brief 생성 자체 실패" 룰 → `non_applicability` 빈 card 가 retrieve top-N 에 들어오면 모든 brief 실패.
- 수정: §9.1/§9.2 출력 schema 에 `avoid_when := card.payload.non_applicability` 매핑 명시. 빈 경우 per-card render `"(none specified)"` fallback (brief 전체 fail 금지).

**F3. atomic write + lock recovery 부재** [Codex adv #1/#8, Codex rev "lock recovery", agy #6]
- 위치: §7.3, §11.1 (config write)
- 문제: card.json / config.yaml / events.jsonl 모두 plain write — disk-full / crash 시 partial file. `mkdir` lock 은 crash 후 영구 보유.
- 수정: §7.3 — 모든 write 는 `<file>.tmp` 작성 → `fsync(file)` → `fsync(dir)` → atomic rename. lock 디렉토리 안에 `{pid, host, created_at}` metadata 저장. audit 단계에 stale-lock detection (created_at > 5min) + `deep-memory-audit --unlock` 명령 추가 (out-of-scope 목록에서 unlock 빼고 MVP 포함).

**F4. concurrent harvest idempotency 부재** [Codex adv #2, Codex rev "Event append not idempotent"]
- 위치: §7.1 (Step A append) + §7.3 (Step C append)
- 문제: 두 CLI 세션 (Claude Code + Codex 다른 터미널) 또는 같은 세션의 retry harvest 가 같은 source artifact 를 두 번 처리 → event 중복 → §6.2 dedupe merge 로 confidence 인플레이션.
- 수정: §7.3 — Step A 의 event append 도 lock 안으로 이동. event key = `sha256(source_path + source_artifact_hash + run_id)`. 중복 key 는 skip. project 단위 harvest lease (`~/.deep-memory/.leases/<project_id>.lease`) 도입.

**F5. LLM 응답 schema validation 부재** [Codex adv #6, Codex rev "§6.1 illustrative"]
- 위치: §7.2 Step B
- 문제: sub-agent 가 parseable but invalid (할루시네이션 필드, 잘못된 enum, 누락 필드) JSON 반환 시 그대로 persist. §6.1 example 만 있고 normative schema 없음.
- 수정: `schemas/memory-card.schema.json` 을 normative 로 격상 (required / enum / format / additionalProperties:false 명시). §7.2 — Step B 출력에 JSON Schema (Ajv) validation. 위반 = candidate fallback.

**F6. pre-LLM redaction gap** [Codex rev #1]
- 위치: §11.1 / §7.2
- 문제: redaction 이 event/card 출력에만 적용. Step B sub-agent 에 보내는 source excerpt 는 redact 안 됨 → 토큰/customer ID 가 LLM 호스트(Claude/Codex/Gemini)에 전송.
- 수정: §11.1 — redaction 적용 순서를 (1) Step A input 단계, (2) Step B input 단계, (3) envelope wrap 직전 3-pass 로 명시. `evidence_paths` 도 home/customer path segment hash 처리.

**F7. score formula scale 불일치** [Codex rev #2]
- 위치: §8 / §10.2
- 문제: `project_sim` 은 0-1 Jaccard, `evidence_quality` 는 `confidence × log(1+n) × (1−r/...)` (무한대 가능), FTS5 `task_sim` BM25 는 무계. 곱셈/가중합 시 BM25 단일 항이 dominance.
- 수정: §8 — 모든 항 0-1 정규화 명시 (BM25 → min-max scaling, evidence_quality → sigmoid clamp). zero-memory / all-stale / missing-profile 케이스 거동 (empty brief vs error) 명시.

**F8. source artifact provenance brittleness** [Codex adv #4]
- 위치: §6.1 envelope.provenance.source_artifacts
- 문제: path-only stored. 원본 artifact rename/delete 시 provenance 깨짐. brief 의 `evidence_paths` 가 dangling.
- 수정: source_artifacts 항목에 `content_hash` (sha256 of artifact at capture time) + `captured_at` + `artifact_kind` 추가. audit 단계에 unresolved-source 리포트.

**F9. privacy global promotion gate 부재** [agy #4]
- 위치: §6.1 / §6.3 / §11
- 문제: card 가 `~/.deep-memory/` 에 저장되는 순간 미래 모든 프로젝트 brief 에 visible. regex redaction 1건 실패 = 모든 미래 프로젝트로 leak.
- 수정: §6.1 payload 에 `privacy_level: local | global` (default local) 추가. validated 상태 전이만으로는 global promote 금지. `deep-memory-audit --promote <id>` 명시 명령 (사용자 review). 글로벌 카드는 `~/.deep-memory/cards/<type>/global/`, 로컬은 `~/.deep-memory/cards/<type>/<project_id>/`.

**F10. memory_root NFS / symlink / read-only preflight 부재** [Codex adv #3]
- 위치: §5 `deep-memory-init` / §7.3
- 문제: lock dir 만들기 직전까지 path 종류 모름. 비싼 harvest 작업 후 lock acquire 실패.
- 수정: §5 init 단계에 realpath resolve + write probe + fsync probe + sqlite lock probe 필수. network mount 감지 시 explicit `--allow-network-root` 요구. read-only 감지 시 brief-only 모드로 degrade.

**F11. cross-runtime sub-agent invocation bridge 부재** [agy #1]
- 위치: §7.2 / D3
- 문제: Claude Code Agent tool vs Codex sub-agent vs Gemini SDK 호출 방식 다름. `agents/memory-distiller.md` 하나로 모든 runtime cover 불가.
- 수정: §7.2 — `scripts/lib/llm-bridge.js` 추상화 도입. 각 runtime adapter (Claude Code: `Agent({subagent_type})` / Codex: `codex review` bash / Gemini SDK: 별도). adapter 부재 시 prompt-based stdin fallback. config.yaml 에 `distill.llm.adapter: auto | claude-agent | codex-bash | gemini-sdk | stdin-fallback`.

**F12. cross-runtime contract tests 부재** [Codex adv #15]
- 위치: §12
- 문제: mock LLM 만 검증. 실제 host adapter 차이가 silent regression 으로 빠짐.
- 수정: §12 — `tests/runtime-contract/` 디렉토리 추가. claude-adapter / codex-adapter / stdin-fallback 별 smoke fixture + schema gate + quality gate (recorded golden response per adapter, hash 비교).

**F13. sqlite Q2 deferred 인데 Phase 4 가 의존** [Opus #4]
- 위치: §17 Q2 / §15 Phase 4
- 문제: `better-sqlite3` (native build) vs `sql.js` (WASM) 선택 미정. Codex 환경에서 native build 가능한지 검증 안 됨. install probe 없으면 Phase 4 시작 시 발견.
- 수정: §17 Q2 → §3 / §15 으로 승격. **`better-sqlite3` pin** + Phase 0 산출물에 `node -e "require('better-sqlite3')"` install probe 추가. Codex sandbox 환경에서 install fail 시 `sql.js` fallback 분기 명시.

**F14. Phase 3 distill 2일 추정 비현실** [Opus #3]
- 위치: §15
- 문제: sub-agent prompt + 5 source type 매핑 + LLM 호출 + timeout/parse fallback + golden eval harness + Step A/B invariant test 모두 2일.
- 수정: §15 — Phase 3 를 Phase 3a (rule mapping + persist, 1.5일) + Phase 3b (sub-agent + eval harness + adapter bridge, 2.5일) 로 분리. 총 ~10.5 영업일.
